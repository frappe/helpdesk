### General code quality guidelines
1. Choose clean code over clever code.
2. Write object oriented code as much as possible.
3. Keep function sizes small, ideally 10 lines.
4. Try and keep files between 100 and 300 lines.
5. Don't keep too many files in a folder or module. Try and keep it under 15.
6. Avoid abbreviations.
7. Use standard API as much as possible.
8. Reuse. Write as little code as possible.
9. Use Frappe UI, espresso for UI styling.
10. Always write tests whenever required, and make sure they work.
11. Build the minimum working app, then iterate towards your goals.
12. Don't hardcode values unless necessary, make them configurable.
13. Document your code, especially the public methods and classes. But avoid obvious comments and seperator comments.
14. No raw SQL, use ORM.
15. If you have multiple functions in a file, the calling function should be on the top and the called functions should be below.
16. All business logic must be API friendly and must be implemented in Server Side.
17. Use tabs for indentation, not spaces.
18. Avoid deprecated API like `$c_obj()`, `cur_frm`, `get_query`, `add_fetch` etc.
19. Don't use frappe.db.commit() or rollback() in your code, let the framework handle it.
20. Always use type hints.

### Pull Requests
1. Always link the issue in the PR description.
2. Always write a clear description of what the PR does, and how to test it.
3. Always create a draft PR, instead of a ready PR
4. Don't mention Claude as co-authors
5. Always use semantic commit messages, and follow the conventional commit format.

### Unit Tests
- Use `bench run-tests --site test-site-ai` to run the tests.
- Be smart about writing tests, avoid writing trivial tests and merge them with other tests if required.
- The tests should cover important functionalities.
- Try and cover edge cases, that's where most of the bugs are.


### Rules for using AI tools like Claude
1. Always review the code generated by AI tools, and make sure it follows the code quality guidelines mentioned above.
2. For research, spawn sub-agents to explore different approaches, and then use the best approach for implementation.
